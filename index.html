The problem

Today

AI agents hold long-lived credentials on cloud servers. If the server is compromised, the agent's authority is compromised. The enforcement layer can grant new permissions, making it a high-value target.

MahaGuardian

Credentials are encrypted and held by the owner's device. Agent rules are cryptographically signed — agents cannot modify or escalate their own permissions. Every action is logged in a tamper-evident audit trail.

Architecture

MahaGuardian's Guardian Daemon runs on the owner's device and manages encrypted credentials, signed agent rules, and a tamper-evident audit log. Agents run on isolated cloud droplets with short-lived, scoped credentials that cannot be escalated.

Owner's device (Guardian Daemon) vault.enc \u2014 age-encrypted credential store SOUL.lock \u2014 ed25519-signed agent rules SOUL.hash \u2014 SHA-256 integrity ledger (signed) audit.log \u2014 append-only, hash-chained | | mTLS v Cloud droplet (one per agent) short-lived credentials (memory only) scoped Guardian Access Tokens partition-blind (cannot see other agents)

A future production architecture extending these guarantees into cloud deployments is under development. See the white paper for design philosophy.

Security properties

E

Encrypted vault

Credentials encrypted with age, wrapped with scrypt-derived AES-256-GCM. Salt and nonce from os.urandom.

S

Signed rules

Agent rules (SOUL.lock) signed with ed25519. Guardian verifies the signature before loading any rule.

A

Audit trail

Append-only SQLite log with hash chaining. Each entry includes SHA-256 of the previous. UPDATE and DELETE blocked.

P

Partition isolation

Each agent runs in its own droplet. Agents cannot see other agents' credentials, rules, or data.

T

Short-lived tokens

Guardian Access Tokens are scoped and time-limited. Agents cannot escalate beyond the token's scope.

I

Input sanitization

Prompt sanitizer strips null bytes, Unicode direction characters, zero-width characters, XML injection, and header injection.

Quick start

# Clone and install git clone https://github.com/partitia-clean/mahaguardian.git cd mahaguardian pip install -r requirements.txt # Initialize the Guardian python -m guardian init # Configure agent rules in SOUL.lock, then sign python -m guardian sign # Deploy an agent python -m guardian deploy --agent research --target hetzner # Run tests (970 passing) pytest

Requires Python 3.11+. Dependencies: pyrage, PyNaCl, cryptography. Source code: partitia-clean/mahaguardian.

Known limitations

We believe in honest disclosure. These are documented limitations of the current V1 implementation.

  • mlock protects raw JSON buffers but not parsed Python dict values. CPython interns strings and GC copies memory.
  • Audit log tail truncation is not detectable by hash chain alone; entry count is checked separately.
  • Guardian must be online for agents to receive new credentials. Agents are always on; Guardian is not.
  • Device loss is total compromise in V1. Multi-device recovery is planned for future versions.
  • No external anchor on the audit log in V1. Future versions will address this.

Roadmap

V1 (current): Laptop-based Guardian Daemon with encrypted vault, signed rules, and tamper-evident audit logging. Functional with 970 passing tests.

Next: Production architecture with stronger credential delegation and independent audit verification. Under development. See the white paper.
