Why this matters now

Through 2025 and into 2026, six disclosed exploits broke production coding agents — Codex, Claude Code, Copilot, and Vertex AI. In every case the attacker did not break the model. They reached the credential the model was holding. A GitHub OAuth token scoped to every authorized repository. A service account that read every bucket in a project. Deny rules silently skipped under context pressure. A branch name carrying a Unicode-obfuscated exfiltration channel.

The common thread: the agent's authority lived in policy, in configuration, or inside the agent's own context window. Each of those is a surface a sufficiently clever prompt can widen, skip, or rewrite.

MahaGuardian moves agent authority out of policy and into cryptography. Rules live in a file signed by the owner's device and verified before any tool call. Tokens are short-lived, narrowly scoped, and issued per task. An agent that should not see the cap table is not asked nicely — it is given a key that does not unlock the cap table.

The shape of the problem

Today

AI agents hold long-lived credentials on cloud servers. If the server is compromised, or the agent is manipulated by hostile input, the agent's full authority is compromised with it.

MahaGuardian

Credentials are encrypted and held by the owner's device. Agent rules are signed by the owner's signing key; an agent cannot modify its own rules without a signing operation it does not control. Every action is logged in a tamper-evident audit trail.

A concrete moment — client boundaries enforced cryptographically

Electric Mobility Co. X is raising a Series B. The founder hires three advisors and gives each one an AI agent. One of those advisors, in their day job, also serves a different client.

Unit Economics Legal Structure Partner Pipeline
Fundraising advisor Allow Allow Allow
Legal advisor Deny Allow Elevate
Advisor's agent (other client) Deny Deny Deny

Three states, not two. Allow means the agent's token unlocks that data class — automatic, no human in the loop. Deny means the token does not reach the data class at all; there is no path for access to occur, and an attacker who compromises the agent cannot widen scope. Elevate means the agent must request one-time human approval before the action proceeds; the request and the approval are both recorded in the audit log.

The hard case is the third row. An advisor who serves both Electric Mobility Co. X and a different client has a separate agent for each engagement. Without enforcement, an agent compromised on one engagement can quietly summarize data from the other — even though no policy explicitly authorized it. MahaGuardian's third row is structurally impossible to bypass: the other-client agent is given keys that do not unlock any of Electric Mobility Co. X's data classes. Compromised agents bypass policy walls; they do not bypass cryptography.

How MahaGuardian holds up against the 2026 attack pattern

The six disclosed exploits map to four defense mechanisms. Each is architectural: it works whether the agent runs in an OpenAI sandbox, an Anthropic environment, or a self-hosted container.

Deny rules skipped under context pressure

Anthropic's Claude Code dropped deny-rule enforcement when token budget tightened. MahaGuardian's rules live in SOUL.lock, outside the agent's context window, signed by the owner's key and verified by the Guardian before every tool call. The agent is not the enforcement layer. It cannot skip what it does not hold.

Permission flips via settings files

Attackers modified .claude/settings.json and .vscode/settings.json on agent hosts to widen scope. MahaGuardian's rules are ed25519-signed; a settings file on a compromised host that lacks a valid signature is rejected at load time. Forging it would require the owner's signing key, which never leaves the owner's device.

Over-broad OAuth tokens

Codex held a GitHub OAuth token scoped to every authorized repository. Vertex AI's service account read every bucket in the project. Guardian Access Tokens in V1 are partition-scoped, time-limited, and cryptographically bound to the agent's mTLS certificate — a token issued to one agent cannot be replayed by another, and access outside the token's partition is denied at the enforcement boundary. Per-task token issuance and minute-scale TTLs are V2 work.

Prompt injection through tool inputs

Jira tickets, branch names, and PR descriptions were used to inject instructions into agents. MahaGuardian does not prevent injection — no layer can. It bounds the consequences to whatever the legitimate scope already permits. An injected instruction to read the cap table fails because the token does not reach the cap table.

Reference: VentureBeat / RSAC 2026 coverage of credential-layer attacks on AI coding agents. Codex, Claude Code, Copilot, and Vertex AI were named in the disclosure series.

How MahaGuardian differs from what exists

Workload identity, secret management, and access brokers all touch this problem. None was designed for an agent that reads arbitrary input and decides what to do next.

The architectural choice that distinguishes MahaGuardian: the signing key that authorizes an agent's rules never leaves the operator's device. There is no centralized cloud broker that can be compromised, subpoenaed, or coerced into widening scope. An attacker who fully owns the agent's execution host still cannot forge a valid rule, because the signature requires a key on hardware the attacker does not control.

Architecture

MahaGuardian's Guardian Daemon runs on the owner's device and manages encrypted credentials, signed agent rules, and a tamper-evident audit log. Agents run on isolated cloud droplets with short-lived, scoped credentials that cannot be escalated.

Owner's device (Guardian Daemon) vault.enc — age-encrypted credential store SOUL.lock — ed25519-signed agent rules SOUL.hash — SHA-256 integrity ledger (signed) audit.log — append-only, hash-chained | | mTLS v Cloud droplet (one per agent) short-lived credentials (memory only) scoped Guardian Access Tokens partition-blind (cannot see other agents)

The trust anchor is the operator's device. Cloud droplets hold no long-term signing material — rotation equals destruction. A future production architecture extending these guarantees to high-availability deployments is under development. See the white paper for design philosophy.

Security properties

E

Encrypted vault

Credentials encrypted with age, wrapped with scrypt-derived AES-256-GCM. Salt and nonce from os.urandom.

S

Signed rules

Agent rules (SOUL.lock) signed with ed25519. Guardian verifies the signature before loading any rule.

A

Audit trail

Append-only SQLite log with hash chaining. Each entry includes SHA-256 of the previous. UPDATE and DELETE blocked.

P

Partition isolation

Each agent runs in its own droplet. Agents cannot see other agents' credentials, rules, or data.

T

Short-lived tokens

Guardian Access Tokens are scoped and time-limited. Agents cannot escalate beyond the token's scope.

I

Input sanitization

Prompt sanitizer strips null bytes, Unicode direction characters, zero-width characters, XML injection, and header injection.

What we do not claim

Security buyers trust founders who name their own threat-model boundaries. These are ours.

  • MahaGuardian does not prevent prompt injection. It bounds the consequences of injection to whatever scope was already authorized.
  • Device loss in V1 is total compromise. Multi-device key recovery is V2 work.
  • Guardian must be online to issue new credentials. Long-running unattended agent fleets are not a V1 use case.
  • Cloud-side scope enforcement requires a proxy boundary between the agent and the application API. Direct SDK calls that bypass the proxy are out of scope for V1.
  • Patent pending is a signal, not a guarantee. Buyers should evaluate the open-source implementation directly.

Known V1 limitations

Honest disclosure of implementation-level limitations.

  • mlock protects raw JSON buffers but not parsed Python dict values. CPython interns strings and GC copies memory.
  • Audit log tail truncation is not detectable by hash chain alone; entry count is checked separately.
  • Guardian must be online for agents to receive new credentials. Agents are always on; Guardian is not.
  • Device loss is total compromise in V1. Multi-device recovery is planned for future versions.
  • No external anchor on the audit log in V1. Future versions will address this.
